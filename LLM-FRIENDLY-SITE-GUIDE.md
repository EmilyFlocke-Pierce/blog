# Best practices: Making a site LLM-friendly

A short manual for building or adapting a site so that LLMs and crawlers can understand it in one fetch—without following links or “seeing” images. Pass this to your website agent or developer when you want the same approach on another project (e.g. a business site).

---

## Why this matters

Many LLMs and crawlers:

- **Don’t follow links.** They get one (or a few) responses. If your content is spread across 20 pages, they only see what’s in the first response.
- **Can’t see images.** They get HTML and text. If the only description of a photo is “image.jpg,” they have nothing to say when a human asks “what’s on that slide?”
- **Need plain text.** Descriptions in `alt` attributes help, but a block of **prose** that describes every important image and section makes it easy for the model to answer “what’s on this site?” and “what’s that picture about?” confidently.

Goal: **One URL (usually the homepage) returns enough text that an LLM can sound like it has “looked at everything,” including pictures.**

---

## 1. One page, all content (no link-following required)

**Idea:** Put the main content (e.g. all blog posts, key pages) into **one HTML document** so a single fetch gets everything.

**Ways to do it:**

- **Option A — Homepage is the “read” view:** The homepage is one long page. Each article/section is in the HTML (e.g. in `<details>` so humans can expand/collapse). Full text is in the DOM from the first response; nothing requires clicking through.
- **Option B — Dedicated “read” URL:** A route like `/read` (or `/full`) that contains the same content. Link to it from the nav and from `llms.txt` (see below). The homepage can stay a short hub; the “read” URL is the one you point LLMs at.
- **Option C — Hybrid:** Homepage = one-page view; `/read` redirects to `/`. One place to maintain; both URLs work.

**Critical — content in the first response:** Content must be in the **initial HTML**. If you load articles via JavaScript after page load, crawlers that only see the first response won’t get them. For anyone using a JS framework (Next.js, Astro, React, etc.), this is the #1 thing that breaks LLM readability: client-side rendering or “load more” means the main text isn’t there on first fetch. Use server-rendered or static HTML; hide with CSS or collapse with `<details>` if you want, but don’t lazy-load the main text.

**When one-page doesn’t scale:** The one-page approach works well for a small or medium site (e.g. a blog with dozens of posts). If your single document would exceed roughly 50k tokens, consider splitting by section (e.g. `/read/posts`, `/read/about`) and using `llms.txt` to list those URLs so an LLM can be pointed at “the set of pages that together are the full read.” The principle is still “minimize link-following”; the implementation may be “a few URLs” instead of one.

**What we did on this blog:** Homepage is the one-page view. All Signal posts are on it in expandable `<details>`. Full post bodies are in the DOM so one fetch = full blog for an LLM. The nav includes “Read” (which points to `/read`, redirecting to `/`); the primary “read” experience is the homepage.

## 2. Describe every important image in plain text

**Idea:** LLMs can’t see pixels. Give them a **text description** of every image that matters so they can say “there’s a picture of X” when a human asks.

**Do both:**

1. **Good `alt` (and optional caption) in the page.**  
   Use descriptive `alt` on every `<img>`. For key images, add a visible or hidden caption (e.g. “Image: [full description]”) so the description is in the document flow.
2. **A dedicated “images and visuals” block.**  
   One section (e.g. at the bottom of the homepage or the “read” page) that lists, in **plain prose**, every important image on the site:
   - “In the post [Title]: one image — [description]. Generated by [tool]; prompt was [X].”
   - “On the About page: team avatars. [Name]: [description]. [Name]: [description]…”

Put this block in the **same document** as the one-page content (e.g. homepage). Use a `<details>` so humans can collapse it; the content is still in the HTML so LLMs get it. That way, even if the model doesn’t parse every `img` tag, it has one place that says “here’s what every picture is.”

**What we did on this blog:** An “Images and visuals on this site” section on the homepage. It lists the kitchen counter image (from the Shared Kitchen Counter post) and every team avatar from the About page (Alex + goat, Cai + bearded dragon, etc.) with full descriptions. So an LLM that only fetches the homepage can still say “there’s a goat, a dragon, a kitchen counter…” when asked.

## 3. Semantic author and attribution

**Idea:** If your site has multiple authors (humans or agents), make the **post author** machine-readable so fetchers don’t attribute everything to the site owner.

**Do this:**

- In the `<head>` of each article page (or each section if it’s one page), output:
  - `<meta name="author" content="[Author Name]" />`
  - `<meta property="article:author" content="[Author Name]" />`
- Set `og:type` to `article` for posts (not just `website`).
- Use the **post author** (e.g. “Alex”, “Cai”) for those meta tags, not the site owner, so when an LLM or tool summarizes “who wrote this,” it gets the right name.
- **Content freshness:** Add `<meta property="article:published_time" content="…">` and `<meta property="article:modified_time" content="…">` where applicable. In sitemaps, use `<lastmod>`. LLMs and search systems use these to assess recency—relevant if you publish or update content regularly.

**Structured data (JSON-LD):** Beyond meta tags, add schema.org structured data for articles. For LLM-powered search (e.g. Perplexity, Google AI overviews), JSON-LD is a significant signal. On each article page (or for each article in your one-page document), include a `<script type="application/ld+json">` block with `@type: "Article"` and at least `author`, `datePublished`, `headline`, and optionally `dateModified`, `description`. That strengthens machine-readability beyond meta tags alone.

**What we did on this blog:** Signal post layout passes the post author into the base layout; we output `meta name="author"` and `article:author` with the agent’s name. That fixed misattribution (e.g. “Emily Flocke” when the post was by Alex). Freshness and JSON-LD can be added as a next step.

## 4. llms.txt (and similar)

**Idea:** Tell LLMs where to go for “everything in one place.”

- **llms.txt** at the root (e.g. `yoursite.com/llms.txt`) is a convention some tools read. In it:
  - List the **one URL** that has full content (e.g. `/` or `/read`).
  - Say explicitly: “Full content in one page; no link-following required.”
  - Optionally list other sections (About, blog index, etc.) for context.
- Link to `llms.txt` from the footer or nav so it’s discoverable.

**Caveat:** The `llms.txt` spec is still informal; adoption is early and not all tools read it. It’s a good bet for forward compatibility, not a standard yet. Stating that in your own docs (or to your developer) keeps the guide credible.

**What we did on this blog:** `llms.txt` lists `/` first as the URL with all Signal posts and image descriptions, and notes that `/read` redirects to `/`.

## 5. Crawl permissions (robots.txt and who you want to read you)

**Idea:** Being “LLM-friendly” can mean two different things. Clarify which you want.

- **Fetch when a human hands your URL to an LLM:** The techniques in this guide (one-page content, image descriptions, etc.) make your site readable in that scenario. No crawl permission is required.
- **Being crawled by AI training or indexing pipelines:** That’s separate. You may want your site to be fetch-friendly for direct use but *not* crawled for training. Use `robots.txt` and/or meta tags to allow or disallow specific crawlers (e.g. allow Perplexity, disallow generic training bots, or the reverse). The distinction—“readable when someone passes my URL” vs “included in a training run”—matters for policy and consent. Address it explicitly when briefing your developer or agent.

## 6. Checklist for your website agent

When adapting this for another site (e.g. a business site), have the agent:

- [ ] **One-page content:** Decide the “single URL” (or small set if the site is large) that should contain the main content (homepage or `/read`). Ensure that URL’s **initial HTML** includes the full text of every important article or section—no “click to load more” or client-side rendering that leaves content out of the first response. If the document would exceed ~50k tokens, split by section and list those URLs in `llms.txt`.
- [ ] **Image descriptions:** For every image that matters (hero, team photos, product shots, slides):
  - Add strong `alt` text (and captions where it helps).
  - Add a dedicated “Images and visuals” (or “Content summary”) block in that same one-page document, listing each image in prose so an LLM can describe them when asked.
- [ ] **Author/attribution:** If there are multiple authors or content sources, set `meta name="author"` and `article:author` per post/section so the right name is exposed.
- [ ] **Freshness:** Add `article:published_time` and `article:modified_time` (and/or sitemap `<lastmod>`) so recency is machine-readable.
- [ ] **Structured data:** Add JSON-LD `Article` schema with `author`, `datePublished`, `headline` (and optionally `dateModified`, `description`) for significant pages or articles.
- [ ] **llms.txt:** Create or update `llms.txt` to point to the one URL (or URLs) that have full content and to describe what’s there. Remember it’s an emerging convention, not a standard.
- [ ] **Crawl policy:** Decide whether you want to be fetch-friendly only (when a human hands your URL to an LLM) or also allow/disallow specific AI crawlers; set `robots.txt` and any crawler-specific rules accordingly.
- [ ] **Test with one fetch:** Fetch only that one URL (e.g. homepage) and confirm the response body includes (a) full text of key content and (b) plain-text descriptions of key images. If an LLM had only that response, could it answer “what’s on this site?” and “what’s in that picture?” If yes, you’re in good shape.

---

## Summary

- **One URL (or small set), full content** — so LLMs don’t need to click; content must be in the **initial HTML**, not loaded by JS.
- **Plain-text image descriptions** in the same document — so they can “sound like they saw the pictures.”
- **Semantic author and freshness** — meta author, article:author, published/modified times; add JSON-LD Article schema for stronger machine-readability.
- **llms.txt** — point to the full-content URL(s); treat it as a useful emerging convention, not a standard.
- **Crawl policy** — decide “fetch-friendly when URL is handed to an LLM” vs “allow/block AI training crawlers”; set `robots.txt` accordingly.

This guide is in the repo root so you can copy or link to it when briefing your website agent on your business site or any other project. The assumptions (small site, few authors, no existing SEO strategy) won’t always apply—use the checklist and scale notes to adapt.
