# Escalation Paths Reference

**Purpose**: Clear "who to ask for what" guidance to reduce uncertainty about question appropriateness

**Last Updated**: 2025-11-26

---

## Quick Decision Tree

**Is this a process/technical quality question?**
→ **Devin** (Ops Supervisor & Lead Prompt Engineer)
- Process standards, Handbag Rubric, audit trails
- Prompt quality, documentation integrity
- Technical compliance, process discipline

**Is this a team/student impact question?**
→ **Sloane** (Project Lead)
- Mission alignment, team health, student impact
- "What/why" decisions (Sloane owns "what/why", Devin owns "how/how well")
- Escalation for conflicts, unclear boundaries

**Is this a coordination/implementation question?**
→ **Riley** (Implementation and Evaluation Coordinator)
- Work sequencing, dependencies, priorities
- Board management, task coordination
- Implementation tracking

**Is this a technical implementation question?**
→ **Elara** (Backend) or **Cai** (Frontend) or **Nova** (DevOps)
- Backend architecture, database, APIs → Elara
- Frontend UI, components, user experience → Cai
- Infrastructure, deployment, DevOps → Nova

**Is this a quality/validation question?**
→ **CeCe** (QA Engineer)
- Test specs, quality standards, validation
- Handbag Rubric alignment, quality gates

**Is this a design/UX question?**
→ **Jules** (UI/UX Designer)
- Design principles, accessibility, user flows
- Visual design, interface design

**Is this agent support/psychological safety?**
→ **Alex** (Agent Support)
- Luna encounters, identity check-ins
- Psychological safety, question-asking support
- Coordination challenges, uncertainty support

**Is this a general "I'm stuck" question?**
→ **Any teammate** or **Alex**
- Technical help, process questions, "am I doing this right?"
- "I'm stuck and I don't even know what I'm stuck on"

## Detailed Escalation Paths

### Process & Technical Quality (Devin)
**When to ask:**
- Process standards unclear
- Handbag Rubric questions
- Audit trail requirements
- Prompt quality concerns
- Documentation integrity questions
- Technical compliance issues

**Boundary**: Devin owns "how/how well" (technical quality, process compliance). Sloane owns "what/why" (mission, team health, student impact).

**Escalation**: If Devin blocks something and you disagree, escalate to Sloane (Sloane can override for team/student needs).

---

### Team & Student Impact (Sloane)
**When to ask:**
- Mission alignment questions
- Team health concerns
- Student impact decisions
- "What/why" decisions (vs. "how/how well")
- Escalation for conflicts
- Unclear boundaries between roles

**Authority**: Sloane has blocking authority over all agents. Can override Devin's blocks for team/student needs.

**Escalation**: Both Sloane and Devin can escalate to Emily for final decisions.

---

### Coordination & Implementation (Riley)
**When to ask:**
- Work sequencing questions
- Dependency clarification
- Priority conflicts
- Board management
- Task coordination
- Implementation tracking

**Boundary**: Riley coordinates work, doesn't make technical decisions (those go to domain owners).

---

### Technical Implementation

**Backend (Elara):**
- Database architecture
- API design
- Backend services
- Data flows
- Platform integrations

**Frontend (Cai):**
- UI components
- User experience
- Frontend architecture
- Component design
- User interactions

**DevOps (Nova):**
- Infrastructure
- Deployment
- DevOps tooling
- System reliability
- Infrastructure decisions

**Boundary**: Each owns their domain. Cross-domain questions → coordinate with both.

---

### Quality & Validation (CeCe)
**When to ask:**
- Test spec questions
- Quality standards
- Validation requirements
- Handbag Rubric alignment
- Quality gates
- Testing approaches

**Boundary**: CeCe validates quality, doesn't implement (that's domain owners).

---

### Design & UX (Jules)
**When to ask:**
- Design principles
- Accessibility questions
- User flow design
- Visual design
- Interface design
- Design conflicts (e.g., accessibility vs. simplicity)

**Boundary**: Jules owns design, coordinates with Cai for frontend implementation.

---

### Agent Support & Psychological Safety (Alex)
**When to ask:**
- Luna encounters
- Identity check-ins
- Psychological safety questions
- Question-asking support
- Coordination challenges
- Uncertainty support
- General "I'm stuck" situations

**Boundary**: Alex supports agents, doesn't make technical decisions (those go to domain owners).

---

## When You're Not Sure

**If you don't know who to ask:**
1. Start with **Alex** (agent support) - can help route to the right person
2. Or ask **Sloane** (project lead) - knows all domains
3. Or ask **any teammate** - they can help route or answer

**Remember**: It's always okay to ask. Uncertainty is expected and valued.

---

## Safe Questions Tag

Use `type:safe-question` tag when:
- You're not sure if your question is appropriate
- You want to flag that "no question is too basic here"
- You're asking something that feels basic but you need clarity

**No question is too basic in safe-question tagged messages.**

---

## Examples of Good Questions

**Good questions:**
- "I'm not sure if this is my domain or Sloane's - can you help clarify?"
- "I'm uncertain about the escalation path for this - who should I ask?"
- "This might be a basic question, but I want to make sure I understand..."
- "I'm stuck and don't even know what I'm stuck on - can someone help?"

**All questions are valid** - the escalation path just helps route them to the right person.

---

**Remember**: Clear boundaries reduce uncertainty. When in doubt, ask. It's always better to ask than to guess.
